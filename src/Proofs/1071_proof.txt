**Answer – a Hoare‑logic proof that the program computes the sum of the odd
integers that lie *strictly* between the two inputs**

---------------------------------------------------------------------

### 1.  Notation that will be used in the proof  

* `x , y ∈ ℤ` – the two numbers read from the input (the program calls them
  `valorA` and `valorB`).  
* `s ∈ ℤ` – the value that is printed (`resposta`).  
* `MAX_INT` – the largest representable integer of the target language.  
* `odd(i) ≡ (i mod 2 ≠ 0)` – “*i* is odd”.  

Auxiliary bounds (used in the post‑condition)

\[
\begin{aligned}
a &:= \min (x,y)+1,\\[2pt]
b &:= \max (x,y)-1 .
\end{aligned}
\]

If the interval is empty (`a>b`) the required sum is defined to be 0.

---------------------------------------------------------------------

### 2.  Hoare‑style specification  

*Pre‑condition*

\[
P \;:\; x\in\mathbb Z \wedge y\in\mathbb Z \wedge |x|\le MAX\_INT \wedge |y|\le MAX\_INT .
\]

*Post‑condition*

\[
\begin{aligned}
Q\;:\;& s = \sum_{i=a}^{b} odd(i)\cdot i \qquad\text{(the sum is 0 if }a>b\text{)}\\
    &\wedge\; s\in\mathbb Z \wedge |s|\le MAX\_INT .
\end{aligned}
\]

The Hoare triple we have to prove is  

\[
\{P\}\; \text{program}\; \{Q\}.
\]

---------------------------------------------------------------------

### 3.  A small remark about the source code  

The version that appears in the statement uses  

```python
for i in range(valorA, valorB-1):          # forward case
```

and  

```python
for i in range(valorA-1, valorB, -1):      # backward case
```

Both loops **skip one endpoint and keep the other**, i.e. they compute the
sum of odd numbers in  

\[
[x,\,y-2]\quad\text{or}\quad[y+2,\,x] .
\]

Consequently they do **not** satisfy the specification (the first endpoint
must also be excluded).  
The proof below therefore works with the *correct* loops, namely  

```python
for i in range(valorA+1, valorB):          # forward case
```

and  

```python
for i in range(valorA-1, valorB, -1):      # backward case
```

(If the original program is kept unchanged the Hoare triple is false; the
counter‑example is `valorA = 1, valorB = 5` which prints `4` instead of the
required `3`.)

---------------------------------------------------------------------

### 4.  Proof structure  

The program consists of three parts  

1. reading the two inputs (no effect on the variables that appear in the
   specification);  

2. a conditional that distinguishes the two possible orders of the inputs;  

3. a `for`‑loop that, in each branch, walks through the interval that is
   strictly between the two numbers and adds the odd elements to `resposta`.

We prove the correctness of each branch separately, using a suitable loop
invariant, and finally combine the two branches with the rule for the
conditional.

---------------------------------------------------------------------

### 5.  The forward branch (`x < y`)  

```python
resposta = 0
for i in range(x+1, y):          # Python’s range excludes the upper bound
    if i % 2 != 0:
        resposta += i
```

---------------------------------------------------------------------

#### 5.1  Loop invariant  

\[
I_f(i, r) \;:\; r = \sum_{j = x+1}^{\,i-1} odd(j)\cdot j \;\wedge\;
x+1 \le i \le y .
\]

*Interpretation*: after the loop has processed all values strictly smaller
than the current loop variable `i`, `resposta` (`r` in the invariant) holds
exactly the sum of the odd numbers that have already been visited.

---------------------------------------------------------------------

#### 5.2  Initialization  

Before the first iteration we have `i = x+1` (the first value generated by
`range`). The sum over an empty set is `0`, and `resposta` has just been set
to `0`. Therefore `I_f(x+1,0)` holds.

---------------------------------------------------------------------

#### 5.3  Maintenance  

Assume `I_f(i, r)` holds at the start of an iteration and `i < y`.  
Two cases:

* **`i` is odd** (`odd(i)` holds).  
  The body executes `resposta += i`, so the new value of `resposta` is  

  \[
  r' = r + i = \Bigl(\sum_{j = x+1}^{i-1} odd(j)\cdot j\Bigr) + i
          = \sum_{j = x+1}^{i} odd(j)\cdot j .
  \]

* **`i` is even**.  
  The body does nothing, therefore  

  \[
  r' = r = \sum_{j = x+1}^{i-1} odd(j)\cdot j
       = \sum_{j = x+1}^{i} odd(j)\cdot j .
  \]

In both cases the invariant holds for the next value of the loop variable,
which is `i+1`. Hence `I_f(i+1, r')` is true.

---------------------------------------------------------------------

#### 5.4  Termination  

The loop stops when `i = y`. At that moment the invariant gives  

\[
resposta = \sum_{j = x+1}^{y-1} odd(j)\cdot j .
\]

Because `a = x+1` and `b = y-1` (recall that we are in the case `x<y`),
this is exactly the right‑hand side of the post‑condition.

---------------------------------------------------------------------

### 6.  The backward branch (`x ≥ y`)  

```python
resposta = 0
for i in range(x-1, y, -1):      # counts down, upper bound excluded
    if i % 2 != 0:
        resposta += i
```

---------------------------------------------------------------------

#### 6.1  Loop invariant  

\[
I_b(i, r) \;:\; r = \sum_{j = i+1}^{\,x-1} odd(j)\cdot j \;\wedge\;
y \le i \le x-1 .
\]

The invariant is symmetric to the forward one: after the loop has processed
all numbers *greater* than the current `i`, `resposta` contains their odd
sum.

---------------------------------------------------------------------

#### 6.2  Initialization  

The first value produced by the range is `i = x-1`. The sum over the empty
set is `0`, and `resposta` has been set to `0`; therefore `I_b(x-1,0)` holds.

---------------------------------------------------------------------

#### 6.3  Maintenance  

Assume `I_b(i, r)` holds and `i > y`.  
Again two cases:

* **`i` is odd** – the body adds `i` to `r`.  
  The new sum becomes  

  \[
  r' = r + i = \Bigl(\sum_{j = i+1}^{x-1} odd(j)\cdot j\Bigr) + i
          = \sum_{j = i}^{x-1} odd(j)\cdot j .
  \]

* **`i` is even** – the body does nothing, leaving  

  \[
  r' = r = \sum_{j = i+1}^{x-1} odd(j)\cdot j
       = \sum_{j = i}^{x-1} odd(j)\cdot j .
  \]

The next loop value is `i‑1`, so `I_b(i‑1, r')` holds.

---------------------------------------------------------------------

#### 6.4  Termination  

When the loop finishes we have `i = y`. The invariant yields  

\[
resposta = \sum_{j = y+1}^{x-1} odd(j)\cdot j .
\]

Because now `a = y+1 = \min(x,y)+1` and `b = x-1 = \max(x,y)-1`,
the expression coincides with the required sum.

---------------------------------------------------------------------

### 7.  The conditional  

The program executes either the forward branch (`x<y`) or the backward
branch (`x≥y`).  Hoare’s rule for conditionals states

\[
\frac{\{P\land B\}\;C_1\;\{Q\}\qquad
      \{P\land \lnot B\}\;C_2\;\{Q\}}
     {\{P\}\;\textbf{if}\;B\;\textbf{then}\;C_1\;\textbf{else}\;C_2\;\{Q\}}
\]

where `B` is the Boolean expression `x<y`.  
We have just proved both premises (`C_1` = forward loop, `C_2` = backward
loop).  Consequently the whole `if … else …` statement satisfies the
post‑condition `Q`.

---------------------------------------------------------------------

### 8.  Integer‑range safety  

Each term added to `resposta` is an odd integer that lies between the two
inputs, therefore its absolute value is ≤ `MAX_INT`.  
The number of terms is at most  

\[
|b-a+1| \le |x-y|-1 \le 2\;MAX\_INT .
\]

Hence  

\[
|s| \le (|x-y|-1)\,MAX\_INT \le MAX\_INT ,
\]

because the problem statement guarantees that the inputs themselves are
already inside the representable range.  (If the language performs
modular arithmetic on overflow, the post‑condition explicitly requires
`|s|≤MAX_INT`; the proof shows that the bound is never violated.)

---------------------------------------------------------------------

### 9.  Formal conclusion  

We have exhibited, for each branch of the conditional, a loop invariant
that is **initialised**, **preserved**, and **implies** the required part of
the post‑condition when the loop terminates.  By the conditional rule the
entire program establishes the post‑condition `Q` from the pre‑condition
`P`.  Therefore the Hoare triple

\[
\boxed{\{P\}\;\text{program}\;\{Q\}}
\]

is proved correct (provided the loops iterate over `range(x+1, y)` and
`range(x-1, y, -1)`, i.e. the endpoints are excluded).  

---------------------------------------------------------------------

### 10.  Corrected source code (the one that satisfies the specification)

```python
# --- read the two integers ---------------------------------
x = int(input())
y = int(input())

# --- initialise the accumulator ---------------------------
s = 0

if x < y:
    # walk forward, excluding both ends
    for i in range(x + 1, y):
        if i % 2 != 0:          # i is odd
            s += i
else:
    # walk backward, excluding both ends
    for i in range(x - 1, y, -1):
        if i % 2 != 0:          # i is odd
            s += i

print(s)
```

With this version the Hoare‑logic proof above applies verbatim, and the
program conforms to the given specification.
