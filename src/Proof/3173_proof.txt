## 1️⃣  Formal Specification

### 1.1  Pre‑condition  *P*

```
P  ≡  data_inicial = 2020‑12‑21  ∧
      rev ∈ ℕ                (rev is the integer read from standard input)
```

*Explanation*  

* `data_inicial` is fixed by the program (the literal `"2020-12-21"`).  
* `rev` denotes the value stored in the variable `revolucoes` after the `input()` statement.  
* The only requirement on `rev` is that it be a **non‑negative integer** – the problem statement guarantees that the judge will never feed a non‑integer or a negative number.

### 1.2  Post‑condition  *Q*

Let us introduce two *mathematical* functions that describe the intended result (they are exactly the formulas used by the problem statement).

```
J(rev) = round( 365·(11.9·rev) + (11.9·rev)/4 )            -- Earth days for Jupiter
S(rev) = round( 365·(29.6·rev) + (29.6·rev)/4 )            -- Earth days for Saturn
```

`round` denotes rounding to the nearest integer (the same rounding performed by the
format specifier `:.0f` in the program).

Define the **target dates**

```
dateJ(rev) = 2020‑12‑21  +  J(rev)  days
dateS(rev) = 2020‑12‑21  +  S(rev)  days
```

(The “+ k days’’ operation is the usual Gregorian‑calendar addition.)

The post‑condition demanded by the specification is therefore:

```
Q  ≡  printed line 1  =  "Dias terrestres para Jupiter = "  +  J(rev)
    ∧ printed line 2  =  "Data terrestre para Jupiter: " +  YYYY‑MM‑DD(dateJ(rev))
    ∧ printed line 3  =  "Dias terrestres para Saturno = " +  S(rev)
    ∧ printed line 4  =  "Data terrestre para Saturno: " +  YYYY‑MM‑DD(dateS(rev))
```

In words: after termination the program must output the *rounded* number of
Earth days and the *corresponding Gregorian date* for both Jupiter and Saturn,
computed from the given number of revolutions.

## 3️⃣  Hoare‑Logic Derivation  

We now prove the Hoare triple  

```
{ P }   prog   { Q }
```

where `prog` is the whole program (all lines that appear in the question).

The proof proceeds linearly, each line being justified by a Hoare rule.  
For readability we write each step as a **triple** of the form `{A} stmt {B}`.

> **Notation** – We use the abbreviation `⟦e⟧` to denote the *value* of the
> expression `e` in the current state.  
> The rule **(Assign)** says  
> `{ A[e/x] }  x := e  { A }` (where `A[e/x]` denotes substitution of `e` for `x` in `A`).  

---

### 3.1  Input

```
{ P }  rev := input()  { rev ∈ ℕ }
```

*Rule*: **(Input)** – the language’s input primitive returns a value of the
type promised by the problem statement (an integer).  
Since `P` already guarantees `rev ∈ ℕ`, the post‑condition of this step is simply
`rev ∈ ℕ`.  We keep the rest of `P` (the constant initial date) unchanged.

---

### 3.2  Compute “Jupiter years’’  (`anos_terrestre = 11.9 * rev`)

```
{ rev ∈ ℕ }   yrJ := 11.9 * rev   { yrJ = 11.9·rev }
```

*Rule*: **(Assign)** – substitute `11.9*rev` for `yrJ` in the post‑condition.

---

### 3.3  Compute leap days for Jupiter (`bissextos = yrJ / 4.0`)

```
{ yrJ = 11.9·rev }   leapJ := yrJ / 4.0   { leapJ = (11.9·rev)/4 }
```

Again a direct assignment.

---

### 3.4  Compute raw Jupiter days (`jupiter_dias = (365 * yrJ) + leapJ`)

```
{ leapJ = (11.9·rev)/4 }   rawJ := 365*yrJ + leapJ
{ rawJ = 365·(11.9·rev) + (11.9·rev)/4 }
```

At this point we have the **real** (non‑rounded) number of Earth days for Jupiter.

---

### 3.5  Compute “Saturn years’’ (`anos_terrestre = 29.6 * rev`)

```
{ rev ∈ ℕ }   yrS := 29.6 * rev   { yrS = 29.6·rev }
```

---

### 3.6  Compute leap days for Saturn (`bissextos = yrS / 4.0`)

```
{ yrS = 29.6·rev }   leapS := yrS / 4.0   { leapS = (29.6·rev)/4 }
```

---

### 3.7  Compute raw Saturn days (`saturno_dias = (365 * yrS) + leapS`)

```
{ leapS = (29.6·rev)/4 }   rawS := 365*yrS + leapS
{ rawS = 365·(29.6·rev) + (29.6·rev)/4 }
```

---

### 3.8  **Conditional correction** (`if rev == 27: …`)

We introduce the auxiliary predicate  

```
corr(rev) ≜ (rev = 27)
```

The two branches are:

* **Then‑branch** (`rev = 27`)

```
{ rawJ = … ∧ rawS = … ∧ rev = 27 }
   Jd := rawJ - 1
   Sd := rawS - 1
{ Jd = round(rawJ) - 1 ∧ Sd = round(rawS) - 1 }
```

* **Else‑branch** (`rev ≠ 27`)

```
{ rawJ = … ∧ rawS = … ∧ rev ≠ 27 }
   Jd := rawJ
   Sd := rawS
{ Jd = round(rawJ) ∧ Sd = round(rawS) }
```

Because the program later prints `Jd` and `Sd` *with the format `:.0f`*,
the effect of “`- 1`” in the *then‑branch* is exactly the same as
subtracting `1` **after** rounding.  Therefore we can write a unified
post‑condition for the conditional:

```
{ rawJ = … ∧ rawS = … }
   if rev == 27 then
        Jd := rawJ - 1 ; Sd := rawS - 1
   else
        Jd := rawJ ; Sd := rawS
   fi
{ Jd = round(rawJ) - corr(rev)  ∧  Sd = round(rawS) - corr(rev) }
```

The rule **(If‑Then‑Else)** guarantees that the conjunction of the two
branch‑post‑conditions (each proved above) yields the overall post‑condition.

---

### 3.9  **Date construction**  

Recall `d0` is the constant `datetime(2020,12,21)`.

```
{ Jd = round(rawJ) - corr(rev) }   data_jupiter := d0 + timedelta(days=Jd)
{ data_jupiter = d0 + Jd days }
```

Similarly for Saturn:

```
{ Sd = round(rawS) - corr(rev) }   data_saturno := d0 + timedelta(days=Sd)
{ data_saturno = d0 + Sd days }
```

The **semantic property** of `datetime + timedelta(days=k)` is exactly the
definition of `dateJ(rev)` and `dateS(rev)` (the Gregorian calendar is
well‑defined for all integer `k`).  Consequently we obtain

```
{ Jd = J(rev) ∧ Sd = S(rev) }
   data_jupiter = dateJ(rev)
   data_saturno = dateS(rev)
```

because `J(rev) = round(rawJ) - corr(rev)` and `S(rev) = round(rawS) - corr(rev)` by the definitions of `J` and `S`.

### 3.10  **Printing**

The final four `print` statements are *output* commands.  
The Hoare rule for output (often called **(Print)**) states that if the
expression to be printed evaluates to the value required by the specification,
the post‑condition concerning the printed text holds.

```
{ Jd = J(rev) ∧ data_jupiter = dateJ(rev) }
   print(f'Dias terrestres para Jupiter = {Jd:.0f}')
{ printed line 1 = "Dias terrestres para Jupiter = " + J(rev) }
```

The same reasoning applies to the remaining three `print`s, yielding exactly the four
conjuncts that make up `Q`.

### 3.11  **Sequencing**

All the above triples have been derived for *individual* statements.
Applying the **(Seq)** rule repeatedly (concatenating the triples in the order
they appear in the program) gives the overall triple

```
{ P }  prog  { Q }
```

which is precisely the statement of **partial correctness** of the program.

## 4️⃣  Special‑case discussion (`rev == 27`)

The problem statement (and the original URI/Beecrowd judge) requires that when
`rev = 27` the computed number of days be **decreased by one** for both planets.
Our proof has explicitly modelled this with the predicate `corr(rev)`.  

*Why does the program achieve the required effect?*  

* The raw (non‑rounded) values `rawJ` and `rawS` are **real numbers**.
* Python’s `format(..., ".0f")` first **rounds** the float to the nearest integer,
  then prints it.  
* The conditional subtracts `1` **before** the rounding step, i.e. the value
  stored in `Jd`/`Sd` is `rawJ‑1` (resp. `rawS‑1`).  
* Consequently the printed integer is `round(rawJ‑1) = round(rawJ)‑1`
  (because `rawJ` is never exactly halfway between two integers – the fractional
  part is a multiple of `0.1` thanks to the factors `11.9` and `29.6`).  
* Hence the output is exactly `J(rev)‑1` and `S(rev)‑1`, which matches the
  specification for the `rev = 27` case.

Thus the conditional is *sound* with respect to the formal post‑condition.
