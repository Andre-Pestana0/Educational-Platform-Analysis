**Answer – Hoare‑logic proof of the given Python program**

Below we give a complete Hoare‑logic proof that the program satisfies the specification that was stated in the problem text:

*For each integer that appears in the input stream the program prints*  

```
"vai ter copa!"   if the integer is 0
"vai ter duas!"   otherwise
```  

*The loop stops exactly when the input reaches EOF.*

The proof follows the usual Hoare‑logic rules (assignment, composition, conditional, while‑loop).  
We first formalise the program, then give the pre‑ and post‑conditions, choose a suitable loop invariant, and finally show that the invariant is established, preserved and that the loop terminates, which together yield the required correctness theorem.



---------------------------------------------------------------------

## 1.  Formalisation of the program

We model the Python program as a small imperative language with the following commands:

```
C ::=  x := e                     (assignment)
     |  if b then C else C       (conditional)
     |  while b do C od          (while‑loop)
     |  try C1 except C2 od      (exception handling)
     |  print s                  (output a constant string)
     |  read(x)                  (read an integer from the input stream,
                                 aborts with an exception EOF if the stream is empty)
```

The concrete program is

```
while true do
    try
        read(reclamacoes) ;
        if (reclamacoes = 0) then
            print "vai ter copa!"
        else
            print "vai ter duas!"
        fi
    except EOF
        break
    endtry
od
```

The only *observable* state components are

* `in` – the (finite) input list of integers,
* `out` – the list of strings that have been printed,
* `i` – the index of the next element of `in` that will be read (initially `0`).

All other variables are local to the program and can be ignored in the specification.

---------------------------------------------------------------------

## 2.  Specification (pre‑ and post‑condition)

```
Pre  :  in = 〈a0 , a1 , … , a{n‑1}〉      (a finite list of integers, n ≥ 0)
       i = 0
       out = 〈 〉                         (empty output list)

Post :  out = 〈 f(a0) , f(a1) , … , f(a{n‑1}) 〉
        where   f(x) = "vai ter copa!"   if x = 0
                     "vai ter duas!"    otherwise
```

In Hoare‑logic notation the required theorem is  

```
{ Pre }  Program  { Post }
```

---------------------------------------------------------------------

## 3.  Loop invariant

The loop processes the input one element at a time, therefore a natural invariant is

```
I  ≡  (0 ≤ i ≤ n)  ∧
      (∀j < i . out[j] = f(in[j]))                (the prefix of length i is already correct)
```

*Intuitively:* after the loop has performed `i` successful iterations, the first `i`
input numbers have been consumed and the corresponding `i` output strings are exactly
the ones required by the specification.

---------------------------------------------------------------------

## 4.  Proof that the invariant holds

We prove the three usual obligations for a while‑loop invariant:

### 4.1  Initialization  

Before the first iteration we have `i = 0` and `out = 〈〉`.  
Thus `0 ≤ i ≤ n` holds and the universal quantifier `∀j < i …` is vacuously true.
Hence `I` holds initially.

### 4.2  Preservation  

Assume `I` holds at the beginning of an iteration **and** the input is *not* EOF.
The body of the loop is the `try … except` construct.  
We analyse the two possible execution paths.

#### 4.2.1  Normal execution (no exception)

* **Statement** `read(reclamacoes)`.  
  By the definition of `read` the next element of `in` (namely `in[i]`) is stored
  in `reclamacoes` and the index `i` is incremented to `i+1`.  
  Formally the command is equivalent to the assignment  

  ```
  reclamacoes := in[i] ;   i := i+1
  ```

* **Conditional** `if (reclamacoes = 0) then … else …`.  

  By the definition of `f` we have  

  ```
  (reclamacoes = 0)  →  print "vai ter copa!"
  (reclamacoes ≠ 0) →  print "vai ter duas!"
  ```

  The command `print s` appends the constant string `s` to `out`.  
  Consequently after the conditional we obtain  

  ```
  out[i‑1] = f(in[i‑1])
  ```

  (the element that has just been processed).

Putting the two steps together we see that after the whole body finishes,
the invariant `I` still holds, now with the index increased from `i` to `i+1`:

```
(∀j < i . out[j] = f(in[j]))   →   (∀j < i+1 . out[j] = f(in[j]))
```

and of course `0 ≤ i+1 ≤ n` because `i < n` (otherwise the input would already be EOF).

#### 4.2.2  Exceptional execution (EOF)

If `read` raises the `EOF` exception, the `except EOF` branch executes the command
`break`.  The `break` terminates the loop *immediately*; no further statements of the
body are executed.  In this case the invariant is *not* required to be preserved,
because the loop will stop and the final post‑condition will be derived from the
invariant together with the negation of the loop guard (see 4.3 below).

Thus, in all normal (non‑exceptional) executions the invariant `I` is preserved.

### 4.3  Termination condition

The rule for a while‑loop with invariant `I` states

```
{ I ∧ B }  C  { I }          (preservation)
{ I ∧ ¬B } → Post           (post‑condition after the loop)
```

Here the loop guard is the constant `true`.  The only way to leave the loop is the
`break` that is executed when the `EOF` exception is raised.  Consequently the
*effective* guard that makes the loop stop is

```
¬B  ≡  EOF has been raised
```

When the loop stops we know that the `read` command could not fetch another element,
hence `i = n` (all input items have been consumed).  Together with the invariant we obtain

```
i = n   ∧   (∀j < n . out[j] = f(in[j]))
```

which is exactly the post‑condition `Post`.  Therefore the third obligation is satisfied.

---------------------------------------------------------------------

## 5.  Termination of the loop

The variable `i` is a natural number that starts at `0` and is increased by `1`
in every *successful* iteration (the ones that do **not** raise EOF).  
Because the input list contains exactly `n` elements, `i` can be increased at most
`n` times; after that the next `read` raises EOF and the loop terminates.
Hence the loop is guaranteed to finish after a finite number of steps, i.e. it
terminates.

---------------------------------------------------------------------

## 6.  Putting everything together

We have shown

* **Initialization:** `Pre ⇒ I`.
* **Preservation:** `{ I ∧ ¬EOF }  Body  { I }`.
* **Exit condition:** `I ∧ EOF  ⇒  Post`.
* **Termination:** the loop cannot execute more than `n` times.

By the Hoare‑logic rule for while‑loops we obtain  

```
{ Pre }  while true do Body od  { Post }
```

which is precisely the required theorem

```
{ in = 〈a0,…,a{n‑1}〉 ∧ i = 0 ∧ out = 〈〉 }
    Program
{ out = 〈 f(a0),…,f(a{n‑1}) 〉 } .
```

Thus the program is **correct** with respect to the given specification.

---------------------------------------------------------------------

### 7.  Summary (in plain English)

* The program reads the input list element by element.
* After each read it prints the correct string according to the value (`0` →
  “vai ter copa!”, otherwise → “vai ter duas!”).
* The loop invariant guarantees that after processing `i` elements the first `i`
  output strings are exactly the ones required.
* The invariant holds initially, it is preserved by each normal iteration,
  and when the input ends (`EOF`) the invariant together with `i = n` yields the
  final required output.
* Because `i` can increase only up to the length of the input, the loop always
  terminates.

Consequently the program satisfies the specification, and the Hoare‑logic proof
above establishes its correctness formally.
