**Answer Overview**

We are asked to *prove* that the Python program shown satisfies a given Hoare‑logic specification.  
The specification (the “pre‑ and post‑conditions”) is the natural one that follows from the statement of the problem:

* **Pre‑condition  \(P\)** – before the program starts  
  *The input file (standard input) consists of an arbitrary finite sequence of lines, each line containing two non‑negative integers `hora` and `minutos` written in the form `hh:mm`. No other characters appear.*

* **Post‑condition  \(Q\)** – after the program finishes  
  *For every line `hh:mm` that was present in the input the program has printed exactly one line  

  \[
  \text{Atraso maximo: } \max\bigl(0,\;(hh\cdot 60+mm)+60-8\cdot 60\bigr)
  \]

  and then terminated (by raising `EOFError`).*

The program is an **infinite loop** that is broken only by the `except EOFError:` clause.  
To prove correctness we will:

1. Write a **loop invariant** \(I\) that captures the “already processed” part of the post‑condition.
2. Show that the invariant holds initially (initialisation).
3. Show that one iteration of the loop preserves the invariant (maintenance).
4. Show that when the loop terminates the invariant together with the termination condition implies the post‑condition (termination).

All of the above is carried out with the standard Hoare‑logic rules (assignment, sequencing, conditional, while‑rule).

---

## 1.  Formalisation of the program

For the purpose of a Hoare proof we abstract the Python constructs that are not essential to the computation:

```text
while true do
    try
        (h,m) := read()          // reads a line “hh:mm”, raises EOFError at end of file
        tot     := h*60 + m + 60
        atraso  := tot - 8*60
        if atraso < 0 then
              out := 0
        else
              out := atraso
        print("Atraso maximo: " ++ out)
    catch EOFError
        break
```

We introduce the following logical variables:

| Symbol | Meaning |
|--------|---------|
| `input` | the (finite) list of pairs \((h_i,m_i)\) that are still unread |
| `output`| the list of numbers already printed (in the same order as the lines that have been read) |
| `h,m`   | the two integers obtained by the current `read` |
| `tot`   | `h*60 + m + 60` |
| `atraso`| `tot - 8*60` |
| `out`   | the value that will be printed in the current iteration |
| `EOF`   | a Boolean that is true iff `input` is empty (i.e. `read` would raise `EOFError`) |

The **semantic effect** of the loop body `S` can be described by the relation  

\[
\begin{aligned}
S\;:\;& (input,output) \longrightarrow
\begin{cases}
\bigl(input',\,output\!\cdot\![\max(0,\;h\cdot60+m+60-8\cdot60)]\bigr) & 
\text{if } input = (h,m)::input' \\
\bigl(\emptyset,\,output\bigr) & \text{if } input = \emptyset
\end{cases}
\end{aligned}
\]

where `output·[x]` denotes list concatenation.

---

## 2.  Loop invariant  

The invariant must express **“everything that has already been processed respects the specification”** and also keep the *remaining* input untouched.  

\[
\boxed{
I\; \equiv\; 
\bigl(\;output = \bigl[\,\max(0,\;h_k\cdot60+m_k+60-8\cdot60)\mid k=1\ldots n\,\bigr]\bigr)
\;\wedge\;
input = \bigl[(h_{n+1},m_{n+1}),\dots,(h_{|input_0|},m_{|input_0|})\bigr]
}
\]

In words:

*`output`* contains, **in order**, the correct “maximum delay” for each line that has already been removed from the original input list `input₀`;  
*`input`* is the suffix of the original input that has **not** yet been read.

Notice that the invariant does **not** mention any program variable that is local to a single iteration (`h,m,tot,…`). This is the usual way to obtain a clean invariant for a *read‑process‑print* loop.

---

## 3.  Proof that the invariant holds initially  

Before the first iteration we have

* `input = input₀` (the whole file)
* `output = []` (nothing printed yet)

Hence the definition of `I` is satisfied with `n = 0`. Formally

\[
\{\,P\,\}\; \textbf{skip} \; \{\,I\,\}
\]

where `skip` is the (implicit) “do‑nothing” that brings us to the entry of the `while`.

Thus the **initialisation** condition of the while‑rule is proved.

---

## 4.  Maintenance – one iteration preserves the invariant  

We must show  

\[
\{\,I \wedge B\,\}\; S \; \{\,I\,\}
\]

where `B` is the loop guard. In our program the guard is the constant `true`; the real “guard” is the *absence* of `EOFError`. In Hoare terms we split the case analysis into the two possibilities that the `try` block either succeeds (`input ≠ ∅`) or raises `EOFError` (`input = ∅`).  

### 4.1  Case `input = ∅` (EOF)  

When `input` is empty the `read` statement raises `EOFError`. Control jumps to the `except` clause, which executes `break`. The loop terminates, and the state after the iteration is exactly the same as before:  

\[
input' = \emptyset,\qquad output' = output .
\]

Since the invariant already states that `output` contains the correct values for the *already processed* lines (there are none left to process), `I` still holds.  

### 4.2  Case `input = (h,m)::input'` (a line is available)  

Assume `I` holds before the iteration, i.e.

\[
output = [\,d_1,\dots,d_n\,],\qquad
d_k = \max\bigl(0,\;h_k\cdot60+m_k+60-8\cdot60\bigr),
\]
and the first unread pair is \((h,m) = (h_{n+1},m_{n+1})\).

The body of the loop performs the following steps (we write them as Hoare triples and compose them with the **sequencing rule**).

| Step | Hoare triple (informal) | Reason |
|------|------------------------|--------|
| 1. `tot := h*60 + m + 60` | \(\{I\}\; tot := h*60+m+60\; \{I\}\) | `tot` is a fresh local variable; it does not affect `I`. |
| 2. `atraso := tot - 8*60` | same argument as (1). |
| 3. `if atraso < 0 then out := 0 else out := atraso` | \(\{I\}\) … `out` gets the value \(\max(0,\;atraso)\) | By the semantics of the conditional we obtain `out = max(0, tot-8*60)`. |
| 4. `print …` | \(\{I\}\) … `output := output·[out]` | The only effect of `print` on the logical state is to append the printed number to the abstract list `output`. |
| 5. `input := input'` | \(\{I\}\) … the unread suffix becomes the tail of the original list | By the definition of `read`, after the successful read the remaining input is exactly `input'`. |

Putting the steps together we obtain the *overall* state after the iteration:

\[
\begin{aligned}
output' &= output \cdot \bigl[\,\max(0,\;h\cdot60+m+60-8\cdot60)\,\bigr] \\
        &= [\,d_1,\dots,d_n,\;d_{n+1}\,] \\
        &\text{with } d_{n+1}= \max(0,\;h_{n+1}\cdot60+m_{n+1}+60-8\cdot60) ,\\[2mm]
input'  &= \text{tail of the original input} 
        = [(h_{n+2},m_{n+2}),\dots] .
\end{aligned}
\]

Thus after the iteration the invariant `I` holds with `n` replaced by `n+1`. Formally,

\[
\{\,I \wedge input\neq\emptyset\,\}\; S\; \{\,I\,\}.
\]

The **maintenance** condition of the while‑rule is satisfied.

---

## 5.  Termination – the loop ends only on EOF  

The loop guard is the constant `true`. In Hoare logic the *while* rule for **partial correctness** does not require a termination argument, but a **total‑correctness** proof does.  

Termination follows from a simple well‑founded measure: the length of the list `input`.  

* **Measure**: `|input|` (non‑negative integer).  
* The body of the loop, when it does **not** raise `EOFError`, removes the first element of `input`, thus decreasing the measure by one.  
* When `|input| = 0` the `read` raises `EOFError` and the loop is exited by the `break`.  

Since the natural numbers are well‑founded, the loop cannot iterate forever. Therefore the program **terminates** for every finite input.

---

## 6.  Deriving the post‑condition  

The while‑rule for total correctness states:

\[
\frac{
   \{P\}\; \textbf{skip}\; \{I\}
   \qquad
   \{I \wedge B\}\; S\; \{I\}
   \qquad
   I \wedge \neg B \;\Longrightarrow\; Q
}{
   \{P\}\; \textbf{while}\; B\; \textbf{do}\; S\; \{Q\}
}
\]

In our case  

* `B` is “the next read does **not** raise `EOFError`”, i.e. `input ≠ ∅`.  
* `¬B` is `input = ∅`.  

When the loop exits we have `I ∧ ¬B`, i.e.

* `input = ∅` (no more lines to read) and  
* `output` contains exactly the list  

  \[
  \bigl[\,\max(0,\;h_k\cdot60+m_k+60-8\cdot60)\mid k=1\ldots |input_0|\,\bigr].
  \]

That is precisely the **post‑condition** \(Q\) required by the specification: one line has been printed for each input line, and each printed value is the correct “maximum delay”.  

Consequently we have proved

\[
\{P\}\; \text{program}\; \{Q\}
\]

— the program is *totally correct* with respect to the given Hoare specification.

---

## 7.  Summary of the proof in Hoare‑logic notation  

| Symbol | Meaning |
|--------|---------|
| `P` | `input = input₀ ∧ output = []` |
| `Q` | `output = [ max(0, h·60+m+60-8·60)  | (h,m)∈input₀ ]` |
| `I` | `output = [ max(0, h·60+m+60-8·60)  | (h,m)∈prefix of input₀ ]  ∧  input = suffix of input₀` |
| `B` | `input ≠ []` |
| `S` | body of the `while` (the `try … except` block) |

Proof obligations  

1. **Initialization** `{P} skip {I}` – holds (empty prefix).  
2. **Maintenance** `{I ∧ B} S {I}` – shown by the case analysis above.  
3. **Termination** `I ∧ ¬B ⇒ Q` – when `input` is empty the invariant tells us that `output` already contains the correct value for every line of the original input.  
4. **Well‑founded measure** `|input|` strictly decreases on each successful iteration, guaranteeing total termination.

All four obligations are discharged, therefore the program satisfies the specification. ∎



---  

### Remarks for readers unfamiliar with Hoare Logic  

*The invariant `I` is the key idea.*  
It tells us “what we already know to be true” at the beginning of each loop iteration. By proving that the loop body does not break that knowledge, we can safely conclude that after the last iteration (when the file ends) the invariant has turned into the desired final property.  

The proof above follows the textbook pattern:

1. **Choose an invariant** that captures the intended relationship between the *processed* part of the input and the *output* produced so far.  
2. **Show it holds initially** (trivial because nothing has been processed yet).  
3. **Show one iteration preserves it** (the arithmetic is straightforward, the conditional implements the `max` function).  
4. **Show that when the loop stops the invariant already *is* the post‑condition** (the remaining input is empty, therefore the processed part is the whole input).  

Because the loop removes exactly one line at each iteration, the natural measure “how many lines are still unread” guarantees that the loop cannot run forever, giving us total correctness.  

Thus the Python program is **correct** with respect to the Hoare‑logic specification.
