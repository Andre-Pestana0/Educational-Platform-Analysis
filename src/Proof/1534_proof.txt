Below is a **complete Hoare‑logic proof** that the program you posted satisfies the informal specification that is usually attached to the problem *“Beecrowd – 1534 – Iniciante – Nível 1”* :

> **Specification**  
> *Pre‑condition* \(P\) : the program is started with the standard‑input stream positioned at the beginning of a (possibly empty) sequence of test‑cases.  
> Each test‑case consists of a single line that contains a **non‑negative integer** `ordem` (the order of the square matrix).  
> The program stops when the input reaches *EOF* (no more test‑cases).  
>  
> *Post‑condition* \(Q\) : for every test‑case read, the program prints an `ordem × ordem` matrix `M` such that  

\[
\forall i,j\;(0\le i,j<\text{ordem})\;:\;
M[i][j]=\begin{cases}
1 & \text{if } i=j\\[2pt]
2 & \text{if } i+j=\text{ordem}-1\\[2pt]
3 & \text{otherwise}
\end{cases}
\]

and the matrix is printed row by row, each row on a separate line, without any extra characters.

The proof is organised as follows  

1.   Formalisation of the program in a tiny imperative language (the “P‑language”).  
2.   Statement of the Hoare triples for each command.  
3.   Loop invariants for the three `for`‑loops that appear in the program.  
4.   Verification that the invariants are **initialised**, **preserved**, and **lead to the desired post‑condition** when the loops terminate.  
5.   Composition of the triples to obtain the overall correctness \(\{P\}\;{\tt program}\;\{Q\}\).

## 1.  Formalisation of the program (P‑language)

We use a very small imperative language that contains the usual constructs:

* **Assignments** `x := e` (where `e` is an arithmetic expression).  
* **Conditional** `if B then C1 else C2`.  
* **While** `while B do C`.  
* **For** loops are encoded as a `while` loop with an auxiliary counter.  

The program, written in this language, is

```
1  while true do                     // outer “read‑until‑EOF” loop
2     try
3        ordem := readInt()          // may raise EOF‑exception
4        // ----- construction of the matrix -----
5        i := 0
6        while i < ordem do
7           j := 0
8           while j < ordem do
9              if i + j = ordem - 1 then
10                M[i][j] := 2
11             else if i = j then
12                M[i][j] := 1
13             else
14                M[i][j] := 3
15             fi
16             j := j + 1
17           od
18           i := i + 1
19        od
20        // ----- printing of the matrix -----
21        i := 0
22        while i < ordem do
23           j := 0
24           while j < ordem do
25              print( M[i][j] )       // without separator
26              j := j + 1
27           od
28           print('\n')
29           i := i + 1
30        od
31     except
32        break                         // leave the outer while‑true
33     endtry
34  od
```

All variables are assumed to be *unbounded* integers, except `M` which is a two‑dimensional array whose size is limited by the current value of `ordem`.  
The primitive `readInt()` returns the next integer from the input stream or raises an *EOF‑exception* if the stream is already exhausted.  
The primitive `print(x)` writes the decimal representation of `x` to the output stream (no spaces, no commas).

## 2.  Hoare triples for the elementary commands

The Hoare rule for **assignment**  

\[
\frac{}{\{\,\phi[e/x]\,\}\;x:=e\;\{\phi\}}
\]

and the rule for **conditional**  

\[
\frac{\{\,\phi\land B\,\}\;C_1\;\{\psi\}\qquad
      \{\,\phi\land\neg B\,\}\;C_2\;\{\psi\}}
     {\{\,\phi\,\}\;{\tt if}\;B\;{\tt then}\;C_1\;{\tt else}\;C_2\;{\tt fi}\;\{\psi\}}
\]

are used throughout the proof; the only non‑trivial part is the verification of the three `while`‑loops (lines 6‑19, 22‑30 and 8‑17).  
For each loop we will give a **loop invariant** \(\mathcal I\) and apply the standard Hoare rule for `while`:

\[
\frac{\{\mathcal I\land B\}\;C\;\{\mathcal I\}}
     {\{\mathcal I\}\;{\tt while}\;B\;{\tt do}\;C\;{\tt od}\;\{\mathcal I\land\neg B\}}
\]

## 3.  Loop invariants

### 3.1  Invariant for the *construction* loop (lines 6‑19)

We have two nested loops:

* Outer loop counter `i` (0 ≤ i ≤ ordem).  
* Inner loop counter `j` (0 ≤ j ≤ ordem).

The invariant must capture the fact that **all entries that have already been visited have already been filled with the correct value**.

> **Invariant 𝓘₁ (outer loop)**  

\[
\mathcal I_1(i) \equiv
\bigl(0\le i\le\text{ordem}\bigr)\;\land\;
\forall p,q\;(0\le p<i\;\land\;0\le q<\text{ordem})\;
   M[p][q]=\operatorname{val}(p,q)
\]

where  

\[
\operatorname{val}(p,q)=
\begin{cases}
1 & p=q\\[2pt]
2 & p+q=\text{ordem}-1\\[2pt]
3 & \text{otherwise}
\end{cases}
\]

> **Invariant 𝓘₂ (inner loop)**  

\[
\mathcal I_2(i,j) \equiv
\bigl(0\le i<\text{ordem}\bigr)\land
\bigl(0\le j\le\text{ordem}\bigr)\land
\forall q\;(0\le q<j)\;
   M[i][q]=\operatorname{val}(i,q)
\]

The two invariants are linked: when the inner loop terminates (`j = ordem`) we obtain exactly the part of the outer invariant that concerns the current row `i`.

### 3.2  Invariant for the *printing* loop (lines 22‑30)

Printing does not modify `M`, it only consumes it. The invariant therefore records how many rows have already been printed.

> **Invariant 𝓘₃**  

\[
\mathcal I_3(i) \equiv
\bigl(0\le i\le\text{ordem}\bigr)\;\land\;
\bigl(\text{rows }0,\dots,i-1\text{ have already been printed}\bigr)
\]

A more precise formulation that is convenient for the inner printing loop is

> **Invariant 𝓘₄ (inner printing loop)**  

\[
\mathcal I_4(i,j) \equiv
\bigl(0\le i<\text{ordem}\bigr)\land
\bigl(0\le j\le\text{ordem}\bigr)\land
\bigl(\text{columns }0,\dots,j-1\text{ of row }i\text{ have already been printed}\bigr)
\]

## 4.  Verification of each loop

### 4.1  Inner construction loop (lines 8‑17)

**Goal**  

\[
\{\mathcal I_2(i,j)\land j<\text{ordem}\}\;
\text{body}
\;\{\mathcal I_2(i,j')\}
\]

where `j' = j+1`.

* **Pre‑condition**: by definition of \(\mathcal I_2\) we already know that for all `q<j` the cell `M[i][q]` contains the correct value.

* **Body** (lines 9‑15) evaluates the three mutually exclusive cases:

  * `i + j = ordem - 1` → `M[i][j] := 2`  
  * `i = j`               → `M[i][j] := 1`  
  * otherwise            → `M[i][j] := 3`

  In each branch the assigned constant coincides exactly with `\operatorname{val}(i,j)`. Hence after the assignment we have  

  \[
  M[i][j]=\operatorname{val}(i,j)
  \]

* **Post‑assignment**: after `j := j + 1` the new value `j'` satisfies `0 ≤ j' ≤ ordem`. Moreover, for every `q < j'` we have either `q < j` (already covered by the induction hypothesis) or `q = j` (the cell we just filled). Consequently  

  \[
  \forall q\;(0\le q<j')\;M[i][q]=\operatorname{val}(i,q)
  \]

  which is exactly \(\mathcal I_2(i,j')\).

Thus the inner loop body **preserves** \(\mathcal I_2\).

* **Termination**: the loop condition is `j < ordem`. When it becomes false we have `j = ordem`. Substituting `j = ordem` in \(\mathcal I_2\) yields  

  \[
  \forall q\;(0\le q<\text{ordem})\;M[i][q]=\operatorname{val}(i,q)
  \]

  i.e. the whole row `i` is correctly filled.

Hence, by the `while` rule we obtain  

\[
\{\mathcal I_2(i,0)\}\;
\text{while }j<\text{ordem}\text{ do }C\;
\{\mathcal I_2(i,\text{ordem})\}
\]

and \(\mathcal I_2(i,0)\) is true because `j` is initialised to `0` (line 9) and the universal quantifier over an empty set is trivially satisfied.

---

### 4.2  Outer construction loop (lines 6‑19)

**Goal**  

\[
\{\mathcal I_1(i)\land i<\text{ordem}\}\;
\text{body (inner loop + }i:=i+1)\;
\{\mathcal I_1(i')\}
\]

where `i' = i+1`.

* **Pre‑condition**: by \(\mathcal I_1\) we know that rows `0 … i‑1` are already correct.

* **Body**:  
  1. Initialise `j := 0` (line 9).  
  2. Execute the inner loop, which we have just proved terminates with the invariant \(\mathcal I_2(i,\text{ordem})\). Hence after the inner loop finishes we have the whole row `i` correctly filled.  
  3. Increment `i := i+1`.

* **Post‑condition**: after the increment we have `0 ≤ i' ≤ ordem`. Moreover, rows `0 … i‑1` are correct (the induction hypothesis) **and** row `i` (the former value of `i`) has just been proved correct. Consequently  

  \[
  \forall p,q\;(0\le p<i'\land0\le q<\text{ordem})\;M[p][q]=\operatorname{val}(p,q)
  \]

  which is exactly \(\mathcal I_1(i')\).

* **Termination**: the loop condition is `i < ordem`. When it becomes false we get `i = ordem`. Substituting in \(\mathcal I_1\) yields  

  \[
  \forall p,q\;(0\le p<\text{ordem}\land0\le q<\text{ordem})\;
        M[p][q]=\operatorname{val}(p,q)
  \]

  i.e. **the entire matrix is correctly constructed**.

Thus the outer construction loop is correct.

---

### 4.3  Inner printing loop (lines 24‑27)

Goal analogous to 4.1 but with invariant \(\mathcal I_4\).  

* The body simply executes `print(M[i][j])` and then `j := j+1`.  
* Since `M[i][j]` already satisfies the specification (proved by the construction loops), the only thing we have to show is that after the increment the invariant still holds: the set of columns already printed grows by exactly one, which is immediate.  

When the loop finishes (`j = ordem`) we have printed **all** entries of row `i` in the correct order.

---

### 4.4  Outer printing loop (lines 22‑30)

Goal analogous to 4.2 but with invariant \(\mathcal I_3\).  

* For each iteration we run the inner printing loop (proved correct) and then output a newline (`print('\n')`).  
* After `i := i+1` we have printed one more complete row, therefore \(\mathcal I_3\) holds for the new value of `i`.  

When the outer loop ends (`i = ordem`) we have printed **all** rows, each followed by a newline, exactly as required by the post‑condition.

---

### 4.5  The outermost `while true` / `try … except` block

The outermost loop (lines 1‑34) is a **read‑until‑EOF** driver. Its correctness follows from the standard Hoare rule for *exception‑controlled loops*:

* The **body** (the `try` block) is executed as long as `readInt()` succeeds.  
* If `readInt()` raises an EOF‑exception, control jumps to the `except` clause, which executes `break`. This terminates the `while true`.  

Because the **pre‑condition** of the whole program does **not** require any particular value of `ordem` before the first test‑case, we can take the trivial pre‑condition  

\[
P\equiv \text{“input stream contains a (possibly empty) sequence of integers”}
\]

and the **post‑condition**  

\[
Q\equiv \text{“for every integer that was read, the corresponding matrix has been printed according to the specification”}
\]

The body of the loop has been proved to transform a *single* integer `ordem` into a correctly printed matrix; the surrounding `while‑true / try‑except` merely repeats this transformation for each integer that exists and stops when there are none. Hence the composition rule for Hoare triples yields  

\[
\{P\}\; \text{program}\; \{Q\}
\]

which is exactly the required **partial correctness** statement.

## 5.  Summary of the proof

| Program fragment | Invariant | Hoare triple (partial correctness) |
|------------------|-----------|--------------------------------------|
| `while j < ordem do …` (inner construction) | \(\mathcal I_2(i,j)\) | \(\{\mathcal I_2(i,0)\}\) `while` \(\{\,\mathcal I_2(i,\text{ordem})\}\) |
| `while i < ordem do …` (outer construction) | \(\mathcal I_1(i)\) | \(\{\mathcal I_1(0)\}\) `while` \(\{\,\mathcal I_1(\text{ordem})\}\) |
| `while j < ordem do …` (inner printing) | \(\mathcal I_4(i,j)\) | \(\{\mathcal I_4(i,0)\}\) `while` \(\{\,\mathcal I_4(i,\text{ordem})\}\) |
| `while i < ordem do …` (outer printing) | \(\mathcal I_3(i)\) | \(\{\mathcal I_3(0)\}\) `while` \(\{\,\mathcal I_3(\text{ordem})\}\) |
| `while true … try … except …` (read‑until‑EOF) | – | \(\{P\}\) `while true` \(\{Q\}\) (by the standard exception‑controlled loop rule) |

All invariants are **initialised** by the `:= 0` statements, **preserved** by the loop bodies (shown in Sections 4.1–4.4), and **strengthen** the required post‑conditions when the loop condition becomes false.  

Consequently the Hoare triple

\[
\boxed{\; \{P\}\; \text{program}\; \{Q\}\;}
\]

holds, i.e. **the program is partially correct** with respect to the given specification.

> **Remark** – The proof above establishes *partial* correctness (if the program terminates, the result satisfies the specification).  
> For this particular program termination is guaranteed: each inner `while` loop increments its counter strictly and is bounded by the non‑negative integer `ordem`; the outer `while true` terminates as soon as the input reaches EOF, which is a finite condition because the input stream is finite. Hence the program is **totally correct**.

### TL;DR (informal recap)

* The program reads an integer `ordem`.  
* It builds a square matrix `M` where each cell gets **1** on the main diagonal, **2** on the anti‑diagonal, **3** elsewhere – proved by the two nested loops using the invariants \(\mathcal I_1,\mathcal I_2\).  
* It then prints the matrix row by row – proved by the two nested loops with invariants \(\mathcal I_3,\mathcal I_4\).  
* The surrounding `while‑true / try‑except` repeatedly applies this construction for every test case until EOF.  

All steps satisfy their Hoare triples, therefore the whole program meets the required pre‑/post‑condition. ∎
