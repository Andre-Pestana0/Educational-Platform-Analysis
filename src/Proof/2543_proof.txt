**Answer – Hoare‑logic proof that the program prints, for every test case, the exact number of
CS‑GO (game = 0) game‑plays that belong to the author whose identifier is given in the case header.**

---

## 1.  The specification

For each *case* the input supplies  

* an integer `identificador` (the author we are interested in) and  
* a finite list `gameplays = ⟨(a₀,j₀), … , (a_{n‑1},j_{n‑1})⟩` of pairs *author, game*.

The required **post‑condition** for a single case is  

\[
\text{POST}\;\equiv\;
\text{output }c\text{ where }c = |\{\,k \mid 0\le k<n,\; a_k = \text{identificador}\land j_k = 0\,\}|
\tag{1}
\]

The **pre‑condition** for a single case is simply that the data are well‑formed:

\[
\text{PRE}\;\equiv\;
\text{identificador}\in\mathbb Z,\;
\text{gameplays}\text{ is a finite list of integer pairs}.
\tag{2}
\]

The whole program processes a *sequence* of cases that the generator `casos()` yields.  
Thus the global specification is  

\[
\{\,\text{the input stream is a concatenation of well‑formed cases}\,\}\;
P\;
\{\,\text{the program prints one line per case that satisfies (1)}\,\}.
\tag{3}
\]

## 2.  Program fragments (P)

```
for identificador, gameplays in casos():            # (A)
    contador = 0                                    # (B)
    for autor, jogo in gameplays:                   # (C)
        if identificador == autor and jogo == 0:    # (D)
            contador = contador + 1                  # (E)
    print(contador)                                 # (F)
```

We prove the correctness of each fragment by Hoare triples of the form  

\[
\{\,\phi\,\}\;S\;\{\,\psi\,\}
\]

where `S` is a statement (or block) and ϕ, ψ are assertions.

---

## 3.  Proof of the **inner** loop (C‑E)

### 3.1 Loop invariant

Let  

\[
I_{\text{inner}}(i) \;\equiv\;
0\le i\le |\,\text{gameplays}\,| \;\land\;
\text{contador}=|\{\,k \mid 0\le k<i,\;
\text{autor}_k = \text{identificador}\land \text{jogo}_k = 0\,\}|
\tag{4}
\]

where `i` denotes the number of already processed elements of `gameplays`.

*Initialization* (`i = 0` before the first iteration)

```
contador = 0
```

- `i = 0` ⇒ the set on the right‑hand side of (4) is empty, therefore its cardinality is 0.
- Hence `contador = 0` establishes `I_inner(0)`.

*Maintenance* – assume `I_inner(i)` holds at the start of an iteration and the next
pair is `(autor_i , jogo_i)`.

1. The test `identificador == autor_i and jogo_i == 0` is evaluated.

   *If the test is false*: `contador` is unchanged. The set of matching indices up to `i+1`
   is the same as up to `i`, thus `contador` still equals its cardinality – `I_inner(i+1)` holds.

   *If the test is true*: `contador` is incremented (`contador := contador + 1`).  
   The set of matching indices up to `i+1` is exactly the set up to `i` plus the new index `i`.  
   Hence the cardinality increases by one, exactly matching the new value of `contador`.  
   Again `I_inner(i+1)` holds.

Thus `I_inner` is preserved.

*Termination* – the loop ends when `i = |gameplays|`.  
By `I_inner(|gameplays|)` we have

\[
\text{contador}=|\{\,k \mid 0\le k<|gameplays|,\;
\text{autor}_k = \text{identificador}\land \text{jogo}_k = 0\,\}|
\]

which is precisely the quantity required by (1) for the current case.

Therefore we have proved the Hoare triple

\[
\{\,\text{contador}=0\;\land\;\text{gameplays is a list}\,\}\;
\text{for autor,jogo in gameplays …}\;
\{\,\text{contador}=|\{k\mid\dots\}|\;\}
\tag{5}
\]

## 4.  Proof of the **outer** loop (A‑F)

### 4.1 Loop invariant

Let the generator `casos()` produce the sequence  

\[
\langle (\text{id}_0,G_0),(\text{id}_1,G_1),\dots,(\text{id}_{m-1},G_{m-1})\rangle .
\]

Define the invariant

\[
I_{\text{outer}}(t) \;\equiv\;
\begin{cases}
0\le t\le m\\[2mm]
\text{the first }t\text{ cases have already been processed and printed correctly, i.e.}\\
\forall\,0\le s<t:\; \text{the line printed for case }s\text{ equals }|\,\{k\mid
(a_{s,k}= \text{id}_s\land j_{s,k}=0)\,\}|
\end{cases}
\tag{6}
\]

### 4.2 Initialization  

Before the first iteration `t = 0`. No case has been processed, so the “for all” clause is vacuously true. Hence `I_outer(0)` holds.

### 4.3 Maintenance  

Assume `I_outer(t)` holds at the beginning of an iteration and the generator yields the next case  
`(identificador, gameplays) = (\text{id}_t, G_t)`.

Inside the body we execute statements (B)–(F).  
By the inner‑loop proof (5) we know that after (C)–(E)

\[
\text{contador}=|\{k\mid (a_{t,k}= \text{id}_t\land j_{t,k}=0)\}|
\tag{7}
\]

The subsequent `print(contador)` therefore outputs exactly the required number for case `t`.  
Consequently, after the body finishes, the first `t+1` cases have been printed correctly, i.e.

\[
I_{\text{outer}}(t+1)
\]

holds. Hence the invariant is preserved.

### 4.4 Termination  

The generator `casos()` terminates because the input stream is finite; consequently the outer `for` loop terminates after `t = m`.  
By `I_outer(m)` we have that **all** `m` cases have been printed with the correct count, which is exactly the global post‑condition (3).

Thus we have proved the Hoare triple for the outer loop

\[
\{\,\text{input stream is a concatenation of well‑formed cases}\,\}\;
\text{for identificador,gameplays in casos(): …}\;
\{\,\text{the program prints one correct line per case}\,\}
\tag{8}
\]

---

## 5.  Whole‑program correctness

The program consists of the sequence  

\[
\underbrace{\text{for identificador,gameplays in casos(): …}}_{\text{outer loop}} .
\]

Applying the **sequence rule** of Hoare logic to (8) yields the global triple (3).  
Both loops are proven to terminate (finite input ⇒ finite number of iterations), so total correctness (partial correctness + termination) follows.

## 7.  Conclusion

Using Hoare logic we have exhibited:

* a **loop invariant** for the inner loop that captures exactly the number of matching
  `(autor, jogo)` pairs processed so far,
* a **loop invariant** for the outer loop that guarantees that every case already processed has been printed correctly,
* a **termination argument** based on the finiteness of the input,
* and by the standard Hoare‑logic inference rules we derived the Hoare triple that matches the given specification.

Hence the program is **totally correct**: for each test case it prints precisely the number of CSGO (game = 0) gameplays authored by the identifier given in the case header. ∎
